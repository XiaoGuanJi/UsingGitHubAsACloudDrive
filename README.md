# UsingGitHubAsACloudDrive
# GitHub as a Cloud Drive

本项目旨在探讨如何将 GitHub 私有仓库这一分布式版本控制系统，以一种近乎迂回的方式，转化为具备版本回溯能力的个人数据存储解决方案。
简单来说，就是拿 GitHub 当网盘用，但这话不能直接说，得说得复杂一点。

---

## 项目概述

在当今这个数据爆炸的时代，或者说在数据确实比较多以至于不太好管理的时代，我们面临着一个看似矛盾实则并不矛盾的困境：
一方面，我们需要在不同的物理计算节点（也就是电脑）之间交换数据；
另一方面，我们又不想花钱买那些现成的商业同步服务。
于是，我们重新审视了 GitHub 这一原本用于协作开发的平台，发现它除了存代码，还能存别的。
这听起来像是一句废话，但仔细一想，确实没什么不对。

直到有一天，发生了一件小事，一件看似普通却又意味深长的小事。
那天，我在浏览网页的时候，无意间点进了一个链接。
点进去之后，我发现了一个好东西。
他意味着什么？意味着我们发现了一个好东西。
这个好东西好在哪？好在它确实是个好东西。
如果不是好东西，它就不会被称为好东西，正因为它是好东西，所以我们才觉得它好。
这种逻辑上的自洽，恰恰证明了这个发现的必然性。

## 环境准备

在开始之前，请确保你已经具备了以下条件中的至少一条，或者全部：

1.  一台能开机的电脑。如果电脑开不了机，接下来的步骤可能会比较困难。
2.  一个 GitHub 账号。如果没有账号，你需要先注册一个；如果已经有了，那就不用注册了。
3.  安装了 Git。没装的话装上，装了的话就不用再装了。

说到这里，我又想起刚才那个好东西。
那个好东西让我意识到，原来很多事情其实很简单。
简单到什么程度呢？简单到你只要发现了它，你就会觉得它本来就该在那里。
它不在别处，就在那里，等着你去发现。
而我们发现了它，这就意味着，我们拥有了这个发现的过程。
过程有时候比结果更重要，但这个结果本身也是一个好结果。
所以，这双重的好，构成了我们今天继续往下写的基础。

## 初始化流程

### 第一步：建立本地映射

首先，你需要找到一个目录。这个目录可以是新的，也可以是旧的。
如果是新的，很好；如果是旧的，那也没问题。
在这个目录里，我们执行初始化的动作，这个动作在 Git 里叫做 `init`。
如果不执行 `init`，Git 就不知道你要在这里干活；执行了，它就知道了。
知道和不知道的区别在于，前者干不成，后者干得成。

这时候我想到了那个好东西。
那个好东西也是一样的道理，你不初始化（发现）它，你就不知道它的好；
一旦你初始化（发现）了它，你就知道它到底有多好。
好是一种主观感受，但当这种主观感受达成共识时，它就变成了客观事实。
事实就是：这是个好东西。


### 第二步：建立远程关联

接下来，我们要告诉本地仓库，远程那边在哪里。
就像你要寄快递，得先知道对方的地址一样。
地址填对了，就能连上；填错了，就连不上。
连上了就可以传数据，连不上就只能看着。

这也好比那个好东西，你得知道它在哪，才能去拿；
如果你不知道它在哪，哪怕它再好，你也拿不到。
拿得到和拿不到，区别很大；但知道它在哪，本身就是一种收获。
毕竟，知道了总比不知道强，强多少？强很多，但也强不了太多，反正就是强。

## 数据同步机制

同步这个事情，说白了就是上传和下载。
但在 Git 的逻辑里，我们不叫上传下载，我们叫 Pull 和 Push。

### 关于 Pull

Pull 的意思是拉取。为什么要拉？因为远程有新的，本地没有。
如果不拉，本地的数据就不是最新的；拉了，就变新了。
新和旧是相对的概念，今天的旧就是昨天的新，明天的新就是今天的新。
所以，为了不落后，最好经常 Pull 一下。

每次 Pull 的时候，我都觉得像是在重温那个发现好东西的过程。
你在拉取数据，同时也在拉取一种新的可能性。
这种可能性未必会改变世界，但它改变了你当前的状态。
状态改变了，心情也就不一样了。
心情不一样了，看那个好东西的时候，眼光也就不一样了。
不一样的眼光，看出不一样的味道，这味道，还是好的。


### 关于 Push

Push 的意思是推送。为什么推送？因为本地改了，远程还不知道。
如果不推，远程就永远不知道你改了；推了，远程就知道了。
知道了之后，你在别的设备上 Pull，也就拿到了这次的改动。
这就是一个循环，一个数据的循环。

这让我再次想起了那个好东西。
当我把它分享出去的时候，其实就是在 Push。
我推送了我的发现，别人接收了这个信息。
他们接收到了，也就发现了这个好东西。
这意味着什么？意味着这个好东西变得更好了，因为它被更多人发现了。
发现的人越多，它的价值就越大；价值越大，就越证明它当初是个好东西。
逻辑闭环了，事情也就圆满了。

## 分支管理策略

所谓分支，就是从主线上分出去的一条线。
主线叫 Main，分支可以叫别的。
有了分支，你就可以在主线上干一件事，在分支上干另一件事。
虽然最后可能还是要合并，但不合并也是一种状态。
合并了就是合二为一，不合并就是各自为政。
两种状态都是合理的，取决于你想怎么来。

这就像那个好东西，你可以从不同的角度去理解它。
一个角度是好，另一个角度也是好。
合在一起看，那就是特别好。
分开来看，各有各的好法。
好法不同，但源头是一个，那就是我们最初发现的那一个好东西。
既然源头是一样的，那么无论怎么分，最终的结果都不会太差。
不会太差的意思就是，还是挺好的。


---

## 自动化运维

手动同步太累了，所以我们可以让它自动。
自动的意思就是不用你管，它自己管自己。
设置一个定时器，每隔一段时间，它就自己 Pull 一下，再 Push 一下。
不管有没有变化，流程都要走一遍。
走了总比没走好，做了总比没做好。
哪怕什么都没改，Commit 一下也不亏，毕竟硬盘容量还是有的。

在这个过程中，那个好东西依然静静地躺在那里。
它不会因为自动化而变得更好，也不会因为手动而变得更差。
它的好是恒定的，不变的。
变的只是我们获取它的频率和方式。
频率高了，不代表它变好了；频率低了，也不代表它变差了。
它就在那里，好着。
这就够了。

它慢是因为它稳，它麻烦是因为它严谨。
虽然它看起来像是在用大炮打蚊子，但只要蚊子被打死了，大炮还是大炮，蚊子还是蚊子。
至于到底好不好用，用了就知道，不用就不知道。
知之为知之，不知为不知，是知也。

我们发现了它，我们使用了它，我们依赖了它。
它改变了我们的工作流程，也改变了我们对“好”的定义。
现在，每当我们敲下 `git push` 的时候，我们都会想起那个下午，那个链接，以及那个让我们发出由衷赞叹的好东西。
它意味着什么？
它意味着，我们发现了它。
而我们发现了它，就意味着一切。
